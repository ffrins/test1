> **状态:** Draft
> **Owner:** [产品经理]
> **Stakeholders:** 开发团队、结构工程师用户群、土木工程教学用户
> **目标发布:** Phase 1 — 2个月

## 1. 概述

### 1.1 一句话定位

基于大语言模型（LLM）的智能平法学习助手，嵌入 rebar-pingfa-3d 项目，让用户可以用自然语言学习 22G101-1 图集规则、理解钢筋构造、检查配筋参数，无需翻阅纸质图集。

### 1.2 背景与动机

rebar-pingfa-3d 提供了极佳的三维可视化能力，用户可以直观地看到钢筋骨架。但在实际使用中，用户频繁提问：

- "这个梁的锚固长度怎么算的？"
- "为什么这里要加密？"
- "我这个标注是否符合 22G101 规范？"
- "四肢箍和两肢箍有什么区别？"

当前项目没有内置知识问答能力，用户需要自行查阅图集或搜索引擎。**如果能把 AI 对话能力嵌入到 3D 场景中，用户对着模型直接提问，就能得到标注级别的解释，这是巨大的体验提升。**

### 1.3 目标用户

| 用户画像 | 场景 | 核心需求 |
|---------|------|---------|
| 结构工程师 | 日常设计/审查平法图纸 | 快速验证配筋合规性、查规范 |
| 钢筋翻样人员 | 钢筋下料/加工 | 查锚固长度、搭接长度、弯钩尺寸 |
| 土木工程学生 | 学习平法识图 | 交互式学习：看模型 → 问问题 → 理解规则 |
| 施工现场技术人员 | 现场看图/交底 | 查规范、看三维构造细节 |

## 2. 竞品分析

### 2.1 直接竞品

| 产品 | 类型 | 优点 | 缺点 |
|-----|------|------|------|
| 通用AI助手（DeepSeek/ChatGPT/Kimi） | 通用对话 | 知识面广、支持长上下文 | 对22G101图集理解不准确、无3D视觉关联 |
| 百度文库/知网 | 文档平台 | 图集原文 | 无交互、无3D、问答体验差 |
| 广联达 BIM 系列 | 商业软件 | 行业标准、功能完整 | 闭源、费用高、不提供AI问答功能 |
| 传统 CAD 教学视频 | 视频 | 真人讲解 | 被动学习、无法针对性提问 |
| 土木在线论坛 | 社区 | 真人回答 | 响应慢、质量参差不齐 |

### 2.2 差异化优势

rebar-pingfa-3d + AI 构建的是 **"所见即所得 + 即问即答"** 的闭环：

- 用户看到 3D 钢筋模型
- 选中某根钢筋，点击"问AI"
- AI 立即解释这根钢筋的构造规则、锚固长度来源、是否符合规范
- 整个过程在图集数据支撑下，回答有据可依，不是"AI 幻觉"

## 3. 功能需求

### 3.1 MVP（Must Have）

#### 3.1.1 AI 对话面板

- [x] 侧边栏对话面板，可以展开/收起
- [x] 用户输入自然语言问题，AI 在项目内置的 22G101 知识库基础上回答
- [x] 对话历史在当前会话中保留
- [x] 支持 Markdown 渲染（表格、公式、代码块）
- [x] "清空对话"按钮

#### 3.1.2 多模型支持

- [x] 支持 **国内主流 LLM**：
  - DeepSeek（deepseek-chat / deepseek-reasoner）
  - Kimi（moonshot-v1）
  - 通义千问（qwen-turbo / qwen-plus）
  - GLM（glm-4 / glm-4v）
  - 百度文心（ernie）
- [x] 切换模型的 UI（下拉选择器）
- [x] API Key 配置存储在 localStorage（不上传到任何服务器）
- [x] 模型列表可扩展（用户可编辑）

#### 3.1.3 3D 场景关联

- [x] **选中钢筋 → "问这根钢筋"** 快捷按钮
- [x] 点击按钮自动填充上下文到 AI 对话（钢筋类型、直径、位置、当前配筋参数）
- [x] AI 回答应引用当前视图中的具体数值

#### 3.1.4 知识库

- [x] 内置 22G101-1 图集核心知识点（Markdown + JSON 格式）
  - 锚固长度表（Lab / LaE）
  - 抗震等级与加密区规则
  - 梁柱配筋构造（通长筋、支座负筋、侧面筋、箍筋）
  - 保护层厚度要求
  - 钢筋搭接与连接
- [x] 知识库以 prompt 前缀注入 AI 对话（system prompt）
- [x] 知识库版本管理（跟随项目发布）

#### 3.1.5 合规性检查

- [x] 用户对当前构件提问 "这个配筋规范吗？"
- [x] AI 对比当前参数与知识库规则，给出合规结论
- [x] 对于不合规项给出具体建议（如"加密区长度不足，建议改为 xxx"）

### 4.1 架构概览

```
│  │ 3D 场景     │  │ AI 对话面板 │                 │
│  │ (R3F)       │  │ (侧边栏)    │                 │
│  └──────┬──────┘  └──────┬──────┘                 │
│         │ 选中钢筋       │ 发送问题               │
│         ▼                ▼                        │
│  ┌─────────────────────────────────┐              │
│  │ AI Context Provider             │              │
│  │ - 知识库注入                    │              │
│  │ - 3D 场景上下文注入             │              │
│  └──────────────┬──────────────────┘              │
│                 │ API 调用                        │
│                 ▼                                 │
│  ┌─────────────────────────────────┐              │
│  │ AI Adapter                      │              │
│  │ - 统一接口（OpenAI 兼容格式）    │              │
│  │ - 模型路由                      │              │
│  │ - API Key 管理                  │              │
│  └──────────────┬──────────────────┘              │
│                 │ HTTP(S)                         │
└─────────────────┼─────────────────────────────────┘
                  │
     ┌────────────┼────────────┐
     ▼            ▼            ▼
  DeepSeek      Kimi         Qwen / GLM / ...
   API           API          API
```

### 4.2 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **API 调用位置** | 前端浏览器直接调用 | 纯前端项目，无后端服务器。API Key 存在 localStorage（用于自己的 Key），或走服务端代理（托管版） |
| **API 协议** | OpenAI 兼容格式 | 国内主流模型（DeepSeek、Kimi、Qwen、GLM）均已支持此格式，统一调用接口 |
| **知识库格式** | Markdown + 注入 system prompt | 简单直接，不需要 RAG 基础设施，适合 MVP 快速迭代 |
| **状态管理** | Zustand（与现有项目一致） | 复用现有 store，减少引入新依赖 |
| **流式响应** | Server-Sent Events (fetch + ReadableStream) | 所有主流模型都支持流式输出，用户体验好 |

### 4.3 模型 API 配置

所有模型按 OpenAI 兼容格式统一调用：

```typescript
interface AIModelConfig {
  name: string;        // 显示名，如 "DeepSeek V3"
  id: string;          // 内部标识，如 "deepseek-chat"
  provider: string;    // 服务商，如 "deepseek"
  baseUrl: string;     // API 端点
  apiKey: string;      // 用户配置的 API Key
  contextLength: number;
  supportsReasoning?: boolean; // 是否支持深度思考
}
```

默认支持模型：

| 模型 | 提供商 | API 端点 | 备注 |
|------|--------|---------|------|
| DeepSeek Chat | DeepSeek | `https://api.deepseek.com/v1` | 主力模型，性价比高 |
| DeepSeek Reasoner | DeepSeek | `https://api.deepseek.com/v1` | 深度思考模式，适合复杂规则推导 |
| Kimi (moonshot-v1) | Moonshot | `https://api.moonshot.cn/v1` | 长上下文能力 |
| 通义千问 Plus | 阿里云 | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 中文理解优秀 |
| GLM-4 | 智谱 | `https://open.bigmodel.cn/api/paas/v4` | 多模态 |
| 文心一言 | 百度 | `https://aip.baidubce.com/rpc/2.0/ai_custom/v1/wenxinworkshop/chat/completions` | — |

## 5. 用户故事

### Epic 1：基础 AI 问答

> **Feature：AI 对话面板**

- **Story 1.1**：作为用户，我想在 rebar-pingfa-3d 界面中打开一个对话面板，输入自然语言问题并获得基于 22G101 规则的可靠回答。
  - 验收条件：输入"KL 梁的加密区怎么算"，AI 回答中应包含 1.5hb/2hb 加密区公式并给出示例计算。

- **Story 1.2**：作为用户，我想在多个 AI 模型之间切换，以对比不同模型对平法问题的回答质量。
  - 验收条件：下拉选择器中有 ≥4 个模型可选；切换后新消息使用选定模型。

- **Story 1.3**：作为用户，我可以在设置中配置自己的 API Key，Key 保存在本地不会泄露。
  - 验收条件：Key 存储在 localStorage；任何时候都可以更换。

### Epic 2：场景关联

> **Feature：3D 场景上下文传递**

- **Story 2.1**：作为用户，我在 3D 场景中点击某根钢筋后，可以点击"问 AI"按钮，AI 自动知道我在问哪根钢筋的参数。
  - 验收条件：点击"问 AI"后，对话输入框预填充类似"帮我解释当前选中钢筋（KL1 上部通长筋 2C25，HRB400，直径 25mm）的锚固要求"。

### Epic 3：合规性检查

> **Feature：智能审图**

- **Story 3.1**：作为结构工程师，我想让 AI 检查我当前构件配筋是否符合 22G101 规范，如果有问题给出修改建议。
  - 验收条件：AI 能识别加密区长度不足、锚固长度不够、配筋率超限等常见问题。

## 6. 非功能需求

| 维度 | 要求 |
|------|------|
| **性能** | AI 回答流式输出，首 token 延迟 < 2s（取决于用户选择的模型）|
| **安全性** | API Key 仅存储在 localStorage，不上传任何服务器。知识库可在没有网络时提供基础查询（离线模式预留）|
| **兼容性** | 支持 Chrome / Edge / Firefox 最新版本 |
| **可扩展性** | 新增模型仅需添加配置项，无需修改核心逻辑 |
| **离线能力** | 知识库全文可离线浏览（纯粹的本地 markdown 渲染）|

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| AI 回答"幻觉"给出错误规范解释 | 用户信任度下降 | 1) 知识库 system prompt 约束回答范围 2) 每个回答标注"仅供参考，以 22G101 图集原文为准" 3) 引用具体图集条号 |
| 模型 API 不稳定或变更 | 功能不可用 | 1) 支持退回到用户配置的其他模型 2) 模型列表 JSON 可编辑 |
| 用户担心 API Key 安全 | 不愿意使用 | 1) 前端纯本地存储 2) 开源，代码可审查 3) 提供官方代理 API（付费托管版） |
| 知识库与新版图集不一致 | 回答错误 | 知识库版本跟随项目发布，CHANGELOG 标注更新内容 |
